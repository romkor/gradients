## Why

The database uses `crypto.randomUUID()` (UUID v4) to generate primary keys for gradients and color stops. UUID v4 is fully random, which causes write-amplification and index fragmentation in SQLite B-tree indexes as rows are inserted out of order. UUID v7 encodes a millisecond-precision timestamp in the leading bits, producing monotonically increasing IDs that keep SQLite inserts sequential and allow sort-by-creation-time directly from the ID.

## What Changes

- Replace `generateId()` in `src/utils/gradient.ts` with a UUID v7 implementation using the `uuid` npm package (`v7()` export)
- Update all call sites that generate IDs for `Gradient` and `ColorStop` objects to use the new function
- Configure `better-auth` to use UUID v7 for all its generated IDs (user, session, account, verification) via `advanced.database.generateId`
- Add `uuid` as a runtime dependency

## Capabilities

### New Capabilities

_(none)_

### Modified Capabilities

- `gradient-persistence`: The `id` column for gradients (and the `id` field of each color stop) will be populated with UUID v7 values instead of UUID v4. Existing rows remain valid — IDs are still stored as `text`. No behavioral requirements change; only the format of newly generated IDs.
- `user-auth`: User, session, account, and verification IDs generated by `better-auth` will be UUID v7 instead of its default random string. No behavioral changes to auth flows; only the format of newly generated IDs.

## Impact

- **`src/utils/gradient.ts`** — `generateId()` updated; all callers pick up the change automatically
- **`src/lib/auth.ts`** — `advanced.database.generateId` set to call `v7()` from `uuid`
- **`package.json` / `pnpm-lock.yaml`** — new `uuid` dependency added
- **Existing data** — no migration needed; old IDs remain valid and readable
- **No API / schema changes** — all `id` columns stay `text`; sort order of new records improves naturally
